# v0.2.0 实现计划：桌面 GUI 与 Word Query

> 本文是 v0.2.0 的历史实施计划，不是当前设计事实源。稳定语义已经收敛到[桌面 GUI 与进程生命周期](../../design/desktop-gui.md)、[Generator 协议与 Word Query](../../design/generator-protocol.md)、[本地 API](../../design/api.md)和 [Anki 同步](../../design/anki-sync.md)；用户流程以 [`product/`](../../product/README.md) 为准。

## 版本目标

当前 v0.1.0 提供可运行的 `vocabd`、本地卡片库、API、预览和 Anki Add-on，但没有面向普通用户的管理入口。v0.2.0 增加一个 Windows-first 桌面 GUI，并用一个内置的无界面 `word-query` generator 组成完整制卡 MVP。

用户在 GUI 中输入一个英语单词。Word Query 使用 DeepSeek 完成两轮、同一上下文内的 LLM 对话：先生成中文解释，再把该解释作为对话上下文追加制卡请求，生成唯一一张 `standard_definition` 候选卡。卡片释义与备注使用中文，例句使用英文。Vocabry 校验并渲染候选卡；用户查看解释及卡片正反面后，明确选择添加或放弃。

本版本交付源码环境下可运行的 MVP。安装器、独立 `.exe`、第三方 generator 安装和跨平台验证不在范围内。

## 产品边界

### GUI 的责任

- 提供唯一的用户界面，第一版只实现纯净的对话式 `chat-v1` 面板。
- 启动、观察并关闭自己拥有的 `vocabd` 子进程。
- 从内置 generator manifest 生成一排入口，并按需启动用户进入的 generator。
- 执行通用输入校验：非空和长度限制；不理解英语单词等业务规则。
- 保存共享 DeepSeek API Key、传递 generator 所声明需要的凭据，并保证凭据不进入日志。
- 展示 generator 的纯文本消息、状态、错误以及 Vocabry 生成的候选卡预览。
- 调用 Vocabry API 完成查重提示和最终 Card 创建；generator 不取得管理 token，也不直接写 SQLite。
- 把添加或放弃的最终结果通知 generator。
- 在首页生成一次性 Anki 配对码，供用户在 Add-on 中手工输入。

### generator 的责任

- 作为没有窗口的独立子进程运行；不导入 GUI 代码，不创建 Qt 控件。
- 通过 stdin/stdout 上的 JSON Lines 协议与 GUI 通信；stdout 只能输出协议消息，日志只能写 stderr。
- 在启动后输出自己的欢迎语、说明和其他自定义文字；这些文字不写入 manifest。
- 校验业务输入，并用普通对话消息引导用户纠正，不把用户输入错误报告为技术故障。
- 管理单个任务的 LLM 对话上下文、冻结可重试请求、生成卡型及结构化字段。
- 不生成 Card HTML，不分配 `card_id`、revision 或 Anki 身份，也不直接提交卡片。

### `vocabd` 的责任

- 继续作为唯一数据一致性边界和 Card 写入者。
- 维护允许的卡型、字段 schema 与 HTML renderer；generator 只能选择已知卡型并填充字段。
- 为尚未持久化的结构化候选卡提供校验和临时预览，且不写数据库。
- GUI 预览与现有 preview 使用同一 renderer、模板和样式，同时静态展示正反面。
- 提供服务身份检查和仅限 admin 的优雅关闭能力。

## 桌面与进程模型

### 技术选择

- Python 3.11+ 与 PySide6。
- 第一版只在 Windows 上定义和验收行为，但避免无必要的 Windows-only UI 设计。
- GUI 与 `vocabd` 是不同进程；每个 generator 也是独立进程。
- 开发环境提供 `vocabry-gui` 启动入口。打包边界应保持清楚，但 v0.2.0 不制作安装器或独立可执行文件。

### GUI 启动

1. 检查配置的本地端口。
2. 如果已有 Vocabry 服务，GUI 拒绝启动，并提示先关闭已有服务。
3. 如果端口被其他程序占用，GUI 用不同信息拒绝启动。
4. GUI 启动自己的 `vocabd` 子进程。
5. 等待健康检查确认服务身份、版本和 API 版本后，才开放 generator 页面。
6. 服务启动失败时保留可诊断错误，不进入降级的无服务模式。

GUI 不连接或接管外部 `vocabd`，也不能按端口号猜测并终止未知进程。

### GUI 关闭

关闭主窗口会进入不可逆的“正在关闭 Vocabry”状态，停止接受新操作：

1. 取消正在执行的 generator 任务。
2. 向已启动的 generator 发送 `shutdown`，等待其退出；必要时终止 GUI 自己启动的子进程。
3. 使用 admin token 调用 `POST /api/v1/admin/shutdown`。
4. 等待 `vocabd` 关闭 WebSocket、数据库和服务进程。
5. 正常关闭超时后，记录非预期故障，明确提示正在强制结束，并终止 GUI 自己启动的 `vocabd` PID。
6. 无论正常还是强制路径，只有确认 `vocabd` 进程确实退出后，GUI 才能退出。
7. 如果强制终止也失败，GUI 保留错误窗口和日志入口，不静默消失。

优雅关闭端点只允许 admin token 调用，只监听本机；Anki token 不具备关闭权限。

### generator 页面生命周期

- manifest 对应 GUI 主界面上的一个 generator 入口。
- 进入页面时按需启动进程，离开页面时关闭；GUI 启动时不预启动所有 generator。
- 空闲时离开直接关闭。任务执行中或存在尚未处理的候选卡时，离开页面前提示结果将被丢弃；确认后取消任务并关闭。
- generator 意外退出时，当前任务作为非预期技术故障结束；GUI 不自动重启它。
- 页面内保留本次会话已完成的对话历史；离开页面后清空，不持久化聊天记录。

## manifest 与启动握手

内置 generator 位于受应用控制的目录，每个 generator 带一份 manifest。v0.2.0 不允许用户安装或注册任意第三方可执行文件。

manifest 至少描述：

```json
{
  "id": "word-query",
  "name": "单词查询",
  "version": "0.2.0",
  "protocol_version": 1,
  "layout": "chat-v1",
  "executable": "...",
  "required_credentials": ["deepseek"]
}
```

manifest 只保存稳定的发现和兼容性元数据。欢迎语、输入提示等 generator 自定义文字由进程启动后通过协议输出，不进入 manifest。

严格握手顺序为：

```text
GUI 启动子进程
→ generator: hello（身份、协议版本、版式）
→ GUI 核对 manifest 与 hello
→ GUI: initialize（任务无关配置和 manifest 声明的凭据）
→ generator: initialized
→ generator: 运行时欢迎消息
→ GUI 开放输入
```

- `hello` 中的 ID、协议和版式必须与 manifest 兼容，否则 GUI 终止进程。
- 初始化完成前不能输出对话消息或接受任务。
- 初始化凭据消息属于敏感数据，不能进入日志、测试输出或异常文本。

## generator 协议 v1

### 传输与基本约束

- GUI 管理子进程，以 UTF-8 JSON Lines 在 stdin/stdout 双向通信。
- GUI 启动子进程时显式设置 UTF-8 Python I/O 环境，generator 入口也重新配置 stdin/stdout/stderr 为 UTF-8；不能依赖 Windows 当前代码页。
- 每行恰好一个完整 JSON object；协议消息不得跨行。
- stdout 禁止裸文本、调试输出或 traceback；stderr 专用于日志。
- 每次任务使用唯一 `task_id`，控制消息还应带可关联的请求标识。
- 未知消息类型、乱序消息、错误 task ID 或 schema 违规均作为协议故障处理。
- 普通对话内容在 v1 中只有 `text`，GUI 按纯文本并保留换行渲染。Markdown 可作为未来显式内容类型加入；当前不能猜测或降级渲染未知类型。
- generator 和 LLM 输出的 HTML 一律不在聊天区域渲染。

协议至少需要表达：

- `hello`、`initialize`、`initialized` 与 `shutdown`；
- 用户输入、任务开始、状态和纯文本对话消息；
- 唯一候选卡及其卡型和结构化字段；
- 取消与 `cancelled`；
- 网络超时、非预期错误和正常完成；
- 超时后的 `retry`；
- GUI 发出的 `candidate_action`：`added` 或 `discarded`。

在实现前先形成机器可校验的消息 schema 和状态转换表。不要只依靠示例 JSON 约定协议。

### 卡片消息边界

候选卡消息只包含 Vocabry 已知的卡型和结构化字段，例如：

```json
{
  "type": "candidate",
  "task_id": "...",
  "card": {
    "card_type": "standard_definition",
    "fields": {
      "word": "concise",
      "phonetic": "/kənˈsaɪs/",
      "definition": "giving information clearly in a few words",
      "example": "Keep your explanation concise.",
      "notes": ""
    }
  }
}
```

Vocabry 白名单管理卡型和字段。未知卡型、未知字段或非法字段必须被拒绝；generator 不能动态提交 renderer 或模板。

用户点击添加后，GUI 先调用 Card API。数据库确认成功后才发送：

```json
{"type":"candidate_action","task_id":"...","action":"added","card_id":"..."}
```

放弃则不调用 Card API，发送 `discarded`。通知 generator 失败不能回滚已经提交的 Card；GUI 必须记录“数据已提交但 generator 状态同步失败”的严重错误并关闭页面，防止状态继续分叉。

## Word Query 工作流

### 单任务状态机

```text
等待输入
→ 校验单词
→ 第一轮：生成中文解释
→ 同时：GUI 展示解释；在同一 LLM 对话追加第二轮制卡请求
→ 生成唯一一张 standard_definition 候选卡
→ Vocabry 校验并渲染正反面
→ 等待用户添加或放弃
→ 清理任务上下文
→ 等待下一个输入
```

关键语义：

- 第一次调用使用中文 prompt 生成中文解释。
- 第二次调用必须把第一次解释作为同一对话的上下文，不能独立重新提示。
- “GUI 展示解释”和“发起第二次调用”可并行，但第二次调用依赖第一轮结果。
- 每个单词建立独立 LLM 对话上下文；新任务绝不能继承上一单词的消息历史。
- LLM 只生成一张候选卡，不生成多个选项。
- 等待用户决定时禁用新输入。用户必须添加或放弃后才能开始新任务。
- Word Query 当前不利用 `candidate_action`，但仍必须接收它并完成通用协议状态转换。

### 输入校验

- GUI 只检查非空和通用长度上限。
- Word Query 判断输入是否符合单个英语单词等业务要求。
- 业务输入不合法时，不调用 LLM；以普通 assistant 文本消息引导用户重新输入。
- 业务输入错误不属于技术故障，也不写 error 级别日志。

### 失败、超时和取消

- 任一 LLM 调用、协议通信、候选校验、HTML 渲染或 GUI 结果处理失败，整个任务失败；不能添加部分结果。
- 普通技术故障显示统一错误提示和可查的 `task_id`，写入脱敏日志，不提供重试按钮。
- API Key 缺失时不启动 generator 任务，GUI 引导用户前往设置。
- Key 是任意文本，GUI 不做格式校验。DeepSeek 拒绝凭据属于非预期技术故障。
- 只有明确的 DeepSeek 网络访问超时提供人工“重试”；不自动重试。
- 每次 HTTP 请求在发送前冻结模型、参数、消息历史和请求体。超时重试必须原样重发被冻结请求：第一轮超时重发第一轮，第二轮超时保留第一轮对话并原样重发第二轮。
- 同一次超时允许用户继续人工重试，直到成功、出现其他错误或取消。界面应提示超时后的重发可能产生重复调用和费用。
- 用户可以取消正在进行的任务。取消是正常操作，只记 info 日志；所有解释、候选和预览结果一并丢弃。
- generator 应响应带 `task_id` 的取消。有限时间内不响应时，GUI 终止该子进程。

### 添加、放弃与同词检查

- 卡片预览完成后同时显示正面和背面，不提供翻面、评分或复习交互。
- GUI 添加前查询未删除 Card。按首尾空格和大小写归一后发现同词时展示已有卡片摘要，并把操作变成需要二次确认的“仍然添加”。
- 同词或完全重复不被系统强制禁止；不自动覆盖、合并或修改旧卡。
- 添加成功后组件标记“已添加”并禁用重复操作；放弃后标记“已放弃”。完成决定后才恢复输入。

## 候选卡预览与 API 变化

v0.2.0 需要补充而非绕过核心服务：

1. 未认证的最小健康检查应能让 GUI 区分 Vocabry、其他端口占用和版本不兼容。
2. 新增 admin-only 的结构化候选校验/临时预览能力。它接受与 Card 创建一致的卡型及字段，调用核心 renderer，但不创建 Card、revision 或 outbox event。
3. 临时预览沿用现有 preview 的 CSP、无权限 iframe 和短时 session 思路，GUI 与浏览器 preview 共享模板和样式。
4. 新增 admin-only shutdown 端点，并保证 Anki token 无权调用。
5. 同词检查优先复用或扩展 Card 查询 API，不允许 GUI 直接打开 SQLite。

具体端点和 schema 在实现时以生成的 OpenAPI 为事实源，并同步更新 `design/api.md`。候选预览必须使用与实际 Card 创建相同的校验和 renderer，避免“预览成功但创建后不同”。

## DeepSeek 凭据与配置

- GUI 管理一个名为 `deepseek` 的共享凭据，多个 generator 可在 manifest 中声明使用它。
- Key 保存于 Windows Credential Manager，不写 JSON、文档、仓库、命令行参数、日志或数据库。
- GUI 只把 manifest 声明需要的凭据通过敏感的 `initialize` 消息交给 generator。
- GUI 允许随时修改或删除 Key。generator 在初始化时取得凭据快照；已启动进程无论忙碌或空闲都继续使用旧 Key，不热更新。
- 修改后只影响新启动的 generator；删除后，已有进程可使用内存中的旧 Key直到退出，之后启动时 GUI 给出未配置引导。
- 设置页明确显示“更改将在下次启动 generator 时生效”。
- GUI 只管理共享 Key。模型名称、API 地址、temperature、prompt 和其他生成策略由各 generator 自己定义，不进入通用 GUI 设置或 manifest。

## 日志与诊断

日志统一位于数据目录并按组件拆分，例如：

```text
.vocabry/logs/
├── gui.log
├── vocabd.log
└── generators/
    └── word-query.log
```

- GUI 捕获子进程 stderr 并写入对应组件日志；不得把 stdout 协议流当日志保存。
- GUI、generator 和服务日志使用同一个 `task_id` 串联单词任务。
- 日志按大小轮转，防止无限增长；GUI 提供“打开日志目录”入口。
- 非预期故障界面显示可回查的任务 ID，不向用户暴露内部 traceback。
- API Key、Authorization header、初始化消息、凭据库返回值和其他已知秘密必须统一脱敏。
- 普通日志不记录完整 LLM 解释和候选卡内容；只记录阶段、状态、耗时、错误类型和脱敏响应摘要。

## 测试策略

本版本决定普通测试直接调用真实 DeepSeek API，不提供模拟 API 路径：

- 测试从 Windows Credential Manager 读取 GUI 保存的共享 Key。
- 未配置 Key 时，普通测试直接失败，并明确提示先通过 GUI 配置；不得跳过并伪装完整通过。
- 断网、额度不足、限流和 DeepSeek 服务故障均会使相关测试失败。这是已接受的稳定性与费用取舍。
- 测试不得断言模型的固定自然语言措辞；应断言响应存在、对话关联正确、第二轮产生唯一合法 `standard_definition` 卡及字段可被 Vocabry 接受。
- 真实端到端测试必须启动 Word Query 子进程并走完整 JSON Lines 握手，断言中文解释、两条中文进度消息以及包含中文或音标 Unicode 的 candidate 能以 UTF-8 到达宿主；直接测试 HTTP client 不能替代这项覆盖。
- 验证第二轮请求确实包含第一轮 assistant 解释及追加的制卡请求。
- 超时重试测试必须证明重发的冻结请求内容完全相同。若真实网络条件无法确定性制造超时，应在不伪造 DeepSeek 响应的前提下，对请求冻结与重发组件做本地确定性测试。
- 测试 Card 写入使用临时数据目录和数据库，不污染正式卡片库。
- 为日志和异常增加泄密回归测试：使用哨兵 Key，断言 GUI 日志、generator 日志、pytest 输出、异常字符串和协议诊断中均不存在该值。

还需覆盖：

- 已有 Vocabry、非 Vocabry 端口占用和服务启动失败；
- 正常关闭、shutdown 超时、强制终止及 GUI 等待 PID 退出；
- manifest/hello 不一致、协议乱序、stdout 污染和 generator 崩溃；
- 页面离开、任务取消、候选未决和凭据修改后的快照行为；
- 候选预览不写数据库，预览与最终创建 renderer 输出一致；
- 同词警告、二次确认、添加幂等性和 `candidate_action` 通知失败；
- shutdown 仅 admin 可用，Anki token 被拒绝。

## v0.2.0 验收标准

版本只有同时满足以下条件才可从 0.1.0 升为 0.2.0：

1. 在 Windows 源码环境运行 `vocabry-gui`，它能启动自己的 `vocabd`，健康后开放界面；端口已占用时可靠拒绝启动。
2. GUI 能从内置 manifest 展示并按需启动 Word Query，完成严格握手；离开页面会关闭进程并清空历史。
3. 用户可在 GUI 中安全保存共享 DeepSeek Key；Key 不进入仓库、配置文件、日志、错误或测试输出。
4. 输入合法英语单词后，真实 DeepSeek 先返回中文解释，随后在同一对话上下文生成唯一合法 `standard_definition` 卡。
5. GUI 展示中文解释，并使用 `vocabd` 的 renderer 同时预览正反面；聊天内容保持纯文本。
6. 用户必须添加或放弃当前候选后才能开始新任务；添加通过 API 写入完整 Card/revision/outbox 闭环，放弃不写数据。
7. 同词卡得到提示但可二次确认添加，不发生隐式合并或覆盖。
8. 普通故障、网络超时、取消和凭据缺失具有约定的不同 UX；只有超时可原样人工重试。
9. GUI 关闭时取消任务、关闭 generator、优雅关闭或强制终止自己的 `vocabd`，并在确认服务 PID 消失后才退出。
10. 普通测试使用真实 DeepSeek Key 并通过，现有 v0.1.0 测试保持通过；Windows 手工验收记录真实运行结果。
11. GUI 能生成 Anki 配对码；使用新数据库重新配对后，Add-on 从 cursor 0 开始且不沿用旧 managed mappings，未消费的创建事件能够补发。

## 明确后置

- Markdown 聊天内容和其他 GUI 版式。
- 一个任务生成多张候选卡或并行处理多个任务。
- generator 自定义 GUI、动态卡型或自带 HTML renderer。
- 第三方 generator 的安装、签名、权限沙箱、更新和卸载。
- 对已启动 generator 热更新 DeepSeek Key。
- 聊天历史持久化。
- Anki 完整模板环境、复习交互或像素级预览兼容。
- Windows 安装器、单文件 `.exe`、自动更新和 macOS/Linux 支持。
