# 可靠性与验证策略

> 文档角色：说明如何证明设计约束成立。具体测试文件和命令在实现后应成为事实源，并回链到不变量。

## 验证原则

- 优先验证用户可观察结果和不变量，不把内部实现形状当作目标。
- 协议使用 schema、golden files 和独立消费者测试，避免只测服务自己的序列化代码。
- 事务与同步必须做故障注入；正常路径通过不能证明崩溃后收敛。
- 真实 Anki、Windows ACL 和打包产物不能完全由 mock 替代。

## 不变量证据矩阵

| 不变量 | 主要证据 |
|---|---|
| `DATA-001` | 包依赖检查、客户端集成测试、数据库进程所有权测试 |
| `IMPORT-001` | 合法/非法 golden jobs、事务回滚、提交前故障注入 |
| `CHANGE-001` | Card/revision/outbox 同事务集成测试 |
| `CARD-001/002` | 状态机、唯一约束、删除与历史测试 |
| `HTML-001/002` | schema、renderer golden tests、手工 HTML 状态机 |
| `API-001` | OpenAPI 契约、并发客户端和 `409` 测试 |
| `SYNC-001..004` | 重复/乱序/断线/漂移/重连测试与真实 Anki 验证 |
| `SEC-001..003` | 鉴权、Host/Origin、路径、注入、session 与 sandbox 测试 |

实现后，表格中的概念描述应补充真实测试路径；不要让本文复制测试用例步骤。

## 测试层次

### 单元与性质测试

- 两种卡型 schema 和确定性 HTML 渲染：转义、空字段、换行和相同输入相同输出。
- revision 状态机：renderer、Anki 手工 HTML、结构化字段再编辑和删除。
- 稳定错误代码、路径边界和资源上限。

### 数据库集成测试

- job 中途错误回滚。
- Card、revision 和 outbox 同事务提交。
- 并发 `expected_revision` 写入只接受一个提交。
- 重启恢复 `processing` job 和未发送事件。
- 迁移前后当前数据、历史和映射不丢失。

### 协议契约测试

- 固定 JSONL、manifest 和 result 合法/非法样例。
- OpenAPI 与独立客户端/Add-on fixture 一致。
- 新旧可选字段、未知协议版本和稳定错误代码。
- 重复事件、乱序 ack、断线重连和过期 revision。

### Anki 与系统测试

- 在支持版本矩阵的最低版本和最新稳定版本创建、更新、删除 Note。
- 用户 HTML 编辑回写；Add-on 自身更新不形成回声循环。
- Anki 离线编辑后重连时保留用户最新操作。
- Windows 安装、单实例、ACL、备份和端到端冒烟测试。

## 故障注入

在以下边界强制终止进程并重启：job 移动后、数据库提交前后、result 写入前后、Anki 应用前后、ack 前后。每个场景都必须证明最终收敛、无漏同步且不重复创建。

测试应记录故障点、提交是否发生和期望恢复动作；不能用固定 sleep 假设异步工作已经完成。

## 备份与恢复

迁移前自动创建 SQLite 一致性备份并保留有限代数。第一版可以不提供恢复命令或界面，但发布前必须有人工可执行、经过演练的恢复步骤。备份存在本身不构成恢复能力。
