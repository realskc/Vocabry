# Vocabry 文档说明

这套文档帮助人类快速理解产品和系统，也为 Coding Agent 提供设计意图与约束。想了解项目本身，请从[项目驾驶舱](PROJECT.md)开始。

## 基本原则

- 文档优先记录代码不容易表达的内容：目标、边界、不变量、决策和取舍。
- 不追求完整复述实现。类、函数、调用链等细节通常直接从代码中调查。
- 同一内容尽量只完整描述一次，其他地方用摘要和链接指向它。
- 人类阅读以建立心智模型和支持判断为主，不需要顺序读完全部文档。
- 改变产品边界、组件职责、数据所有权、外部协议或关键取舍的上层调整，必须同步新增或更新 ADR；普通实现细节和内部重构不需要记录为决策。
- 具体文档形式不预先锁死，可在工程实践中根据实际效果调整。

## 推荐人类阅读

这里只列出完全为人类理解和决策准备的文档：

1. [项目驾驶舱](PROJECT.md)：产品、核心路径、系统摘要和当前问题。
2. [产品范围](product/scope.md)：第一版做什么、不做什么。
3. [系统地图](architecture/system-map.md)：组件关系、数据流和依赖方向。
4. [ADR 索引](decisions/README.md)：关键选择、代价和重新评估条件。
5. [实施路线图](roadmap/implementation-plan.md)：阶段目标、先后关系和当前推进方向。

其余文档主要服务于具体设计、实现和验证任务，不属于人类了解项目时的默认阅读材料。

## 文档与代码

文档主要表达预期行为和设计意图，代码与运行结果表达当前实际行为。两者可能暂时不同步；本项目承认这种风险，但应尽量减少它：

- 改变产品行为、组件边界、协议或不变量时，顺手检查相关文档。
- 发现差异时，判断是实现缺陷、文档过期还是迁移中的临时状态。
- 暂时无法解决的差异应明确记录，避免被误认为稳定设计。

当前不建设从代码自动生成事实文档的系统，也不要求文档与代码不一致在机制上完全不可能发生。代码、测试、文档和人工判断共同构成项目知识；具体同步方法可以随着工程实践自然演化。
