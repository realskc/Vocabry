# Vocabry 产品介绍

Vocabry 是一个运行在用户电脑上的英语词汇卡片库。它负责接收、保存、修改和追踪卡片内容，再把卡片交给 Anki 复习。Anki 仍然负责排程、复习界面和学习记录；Vocabry 不试图替代它。

Vocabry 解决的问题不是“怎样背单词”，而是“怎样长期管理由工具生成、又允许用户手工修改的词汇卡片”。当卡片来自脚本、Coding Agent、词典或未来的 LLM 工具时，直接把结果写入 Anki 会带来几个困难：生成器必须理解 Anki、内容缺少独立历史、批量失败容易留下半成品，而且 Anki 中的人工修改很难与结构化内容共存。Vocabry 在生成工具与 Anki 之间增加一个本地、可检查的内容层。

## 谁会使用它

第一版服务于同一台 Windows 电脑上的单个用户：

- 用户在 Anki 中复习并偶尔手工调整卡面；
- 外部生成器批量产生词汇卡片；
- Coding Agent、脚本或 API 工具管理卡片；
- `vocabd` 在本机保存数据并协调这些参与者。

目前没有面向普通用户的完整管理界面。需要创建、查询或编辑卡片时，使用本地 API；浏览器页面只负责预览。这个产品形态适合愿意使用脚本或 Agent 的早期用户，还不是开箱即用的桌面应用。

## 一次完整使用过程

1. 生成器把一批卡片写成一个本地导入任务。
2. 用户或管理工具请求 `vocabd` 导入任务。
3. `vocabd` 先检查整批内容；只要有一张卡不合法，整批都不进入卡片库。
4. 合法卡片获得稳定身份、HTML 卡面和第一条历史版本。
5. Anki Add-on 与 `vocabd` 配对后，把这些卡片创建为 Anki Note。
6. 用户照常在 Anki 中复习，也可以修改受管 Note 的正反面 HTML。
7. Add-on 把修改或删除回传给 Vocabry；Vocabry 保存新的版本，并继续向 Anki 发送后续变化。

服务离线时不影响 Anki 自身的复习功能，但同步能力会暂停。当前 Add-on 的离线变更保全与重连对账仍不完整，具体限制见[当前状态](../STATUS.md)。

## 产品能力

当前代码提供：

- SQLite 本地卡片库；
- 两种内置词汇卡型；
- 卡片的创建、查询、编辑、软删除和版本历史；
- 将纯文本字段安全、确定性地渲染成 HTML；
- 原子批量文件导入和机器可读错误报告；
- 只监听本机的 HTTP API 与 WebSocket 事件流；
- 有时限的一次性 Anki 配对码；
- 有时限、只读且用 sandbox iframe 隔离的浏览器预览；
- 一个可安装的 Anki Add-on 原型。

## 产品边界

第一版有意保持小范围：单用户、单机、单 SQLite 数据库和单 Anki Profile。它只处理纯文本词汇字段，不包含图片、音频、Markdown、自定义卡型、自动去重或卡片合并。

Vocabry 不包含：

- 复习算法、学习进度或复习界面；
- LLM、词典、网页抓取或其他内容生成能力；
- 多用户、多设备或云同步；
- 完整管理 GUI 或管理 CLI；
- 版本恢复界面；
- 成熟的 Windows 安装器、后台服务和自动备份。

这些不是隐藏在文档里的待办承诺。只有真实需求出现后，才应重新判断是否进入产品。

## 内容所有权

Vocabry Card 是内容主记录，Anki Note 是复习端副本。这样可以独立保存结构化字段、生成来源和版本历史，同时允许 Anki 用户修改当前卡面。

“内容主记录”不意味着 Anki 的人工修改会被忽略。Anki 修改正反面后，该 HTML 会成为 Vocabry 当前版本；之后若结构化字段再次变化，Vocabry 会重新渲染卡面并覆盖先前的手工 HTML。这个取舍需要用户理解，详见[核心概念](concepts.md)。

## 成功标准

对第一版而言，产品真正可用意味着：

- 一批卡片要么完整导入，要么一张都不导入；
- 用户能看懂错误发生在哪个文件、行和字段；
- 本地卡片变化最终准确反映到 Anki；
- Anki 中的受管卡面修改和明确删除最终准确反映到本地；
- 重复消息、断线和进程重启不会重复创建 Note 或永久漏掉变化；
- 未授权进程和网页不能读取或修改卡片。

仓库已经实现其中的核心路径，但尚未为真实 Anki、崩溃恢复和 Windows 分发提供充分证据。因此当前状态是“可运行 MVP”，不是可发布成品。
